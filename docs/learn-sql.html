What is SQL?

SQL (Structured Query Language) is how we talk to databases. A database stores information in tables — like a spreadsheet with rows and columns.

We have a table called students with these columns:

id name age grade city

1. SELECT — get data

The * means "all columns".

SELECT * FROM students;

2. Pick only some columns

Ask for just the columns you want.

SELECT name, age FROM students;

3. WHERE — filter rows

Only show rows that match a condition.

SELECT name, age FROM students WHERE age > 12;

4. ORDER BY — sort results

Sort by age, biggest first (DESC).

SELECT name, age FROM students ORDER BY age DESC;

5. COUNT — how many?

Count how many students are in each grade.

SELECT grade, COUNT(*) AS total FROM students GROUP BY grade;

6. INSERT — add a new row

Add a brand new student, then run SELECT * FROM students; to see them.

INSERT INTO students (name, age, grade, city) VALUES ('Zara', 11, 6, 'Lagos');

7. UPDATE — change existing rows

Change data that is already there. Use WHERE to pick which rows! Then run SELECT * FROM students; to see the change.

UPDATE students SET city = 'Jos' WHERE name = 'Ben';

8. DELETE — remove rows

Delete rows that match a condition. ⚠️ Always use WHERE — without it, DELETE removes every row! (Click Reset data to get them back.)

DELETE FROM students WHERE name = 'David';

9. JOIN — combine two tables

We also have a teachers table that lists the teacher for each grade:

grade teacher

A JOIN matches rows from both tables where the grade is the same — so each student gets their teacher's name:

SELECT students.name, students.grade, teachers.teacher
FROM students
JOIN teachers ON students.grade = teachers.grade;

10. CREATE TABLE — make your own table

Before you can store data, you build the table and say what columns it has and what type each one is (TEXT for words, INTEGER for whole numbers).

CREATE TABLE clubs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT
);

11. PRIMARY KEY & FOREIGN KEY

These two ideas are how tables stay organized and connected.

🔑 PRIMARY KEY — a column that gives every row a unique id. No two rows can share it. In our students table, id is the primary key: student #1 is always student #1. AUTOINCREMENT just means the database fills in the next number (1, 2, 3, …) automatically.

🔗 FOREIGN KEY — a column that points to the primary key of another table. It's how two tables are linked. For example, an enrollments table can use a student_id column that refers back to students.id:

CREATE TABLE enrollments (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  student_id INTEGER,
  subject TEXT,
  FOREIGN KEY (student_id) REFERENCES students(id)
);

Now student_id in enrollments must match a real id in students — so the data can never point to a student who doesn't exist. This button builds the table, signs up student #1 for two subjects, and joins it back to students so you can see the link in action:

In short: a primary key identifies a row in its own table; a foreign key borrows that id to link to it from another table.

📝 Final Exercise: Build the "My Diary" Database

Now put it all together! A diary app needs to remember two things: who the users are, and what entries each user wrote. You'll build both tables and link them with a foreign key — exactly like the lessons above.

The plan: one user can write many entries, so each entry must remember which user it belongs to. We do that with a user_id foreign key on the entries table.

users( id 🔑, username, email )
entries( id 🔑, user_id 🔗→ users.id, mood, title, body, entry_date )

Task 1 — Create the users table.

It needs: an id as the PRIMARY KEY (auto-incrementing), a username (TEXT) and an email (TEXT).

💡 Hint 
CREATE TABLE name ( id INTEGER PRIMARY KEY AUTOINCREMENT, ... );
✅ Show solution 
CREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT,
  email TEXT
);

Task 2 — Create the entries table with a foreign key.

It needs: id (PRIMARY KEY), user_id (the FOREIGN KEY pointing to users.id), plus mood, title, body and entry_date.

💡 Hint 
... FOREIGN KEY (user_id) REFERENCES users(id)
✅ Show solution 
CREATE TABLE entries (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER,
  mood TEXT,
  title TEXT,
  body TEXT,
  entry_date TEXT,
  FOREIGN KEY (user_id) REFERENCES users(id)
);

Task 3 — Add a user, then add two diary entries for that user.

Insert one user (their id will be 1), then insert two entries that use user_id = 1.

✅ Show solution 
INSERT INTO users (username, email)
VALUES ('seun', 'seun@example.com');

INSERT INTO entries (user_id, mood, title, body, entry_date)
VALUES
  (1, '😊', 'A calm start', 'Made coffee before the house got busy.', '2026-06-30'),
  (1, '🥰', 'Dinner with friends', 'We laughed about old times.', '2026-06-29');

Task 4 — Show each entry next to its author's username (JOIN).

Join entries to users on the matching id, and show the username, title and mood.

✅ Show solution 
SELECT users.username, entries.mood, entries.title
FROM entries
JOIN users ON entries.user_id = users.id;

🎉 If Task 4 shows each entry beside seun, you've built a real relational database — the same shape that powers the My Diary app!
